Maven¶

Important

Pour la partie industrialisation des processus de test, nous nous baserons sur le projet Pizza-Spring. Vous devez télécharger le code source ici :

pizza-spring.zip

Si vous voulez lancer l’application, pensez qu’elle nécessite une base de données. Le schéma MySQL se trouve dans src/main/sql/db_pizza.sql et le fichier de configuration pour l’accès à la base de données est src/main/resources/jdbc.properties.

Dans ce chapitre, nous allons voir comment configurer des plugins avec Maven pour une application Java. L’utilisation de plugins permet d’enrichir les tâches automatiques à réaliser lors de la construction de notre projet. Ainsi, nous allons pouvoir, non seulement exécuter les tests automatisés, mais également produire des rapports et des métriques à partir de l’exécution des tests ou encore déployer notre application dans Tomcat.

Utilisation de Maven en ligne de commandes¶

Si Maven est un outil intégré aux environnements de développement Java (Eclipse, IntelliJ, NetBeans), il s’agit néanmoins d’un outil autonome qu’il est possible d’exécuter depuis la ligne de commandes. Nous allons voir comment nous affranchir des environnements de développement et comment appeler directement Maven. En ligne de commandes, le programme Maven s’appelle : mvn

Exercice¶

Tester Maven en ligne de commandes

Pour Windows

  1. Téléchargez l’archive zip de Maven

  2. Dézippez l’archive sur votre disque (par exemple dans c:\Program Files)

  3. Dans les variables d’environnement du système, ajoutez les variables suivantes :

    M2_HOME

    Cette variable d’environnement doit indiquer le chemin du répertoire dézippé.

    PATH

    Modifiez cette variable d’environnement pour ajouter %M2_HOME%\bin dans la liste des chemins des applications.

Pour MacOS

  1. Téléchargez l’archive tar.gz de Maven

  2. Dézippez l’archive dans le répertoire de téléchargement

  3. Dans un terminal, déplacez le répertoire créé dans le répertoire des applications :

    mv Downloads/apache-maven-3.6.3 /Applications/

  4. Créez si nécessaire un fichier .bash_profile et éditez ce fichier :

    touch .bash_profile
open -e .bash_profile

  5. À la fin du fichier, ajoutez la déclaration des variables d’environnement M2_HOME et PATH :

    export M2_HOME=/Applications/apache-maven-3.6.3
export PATH=$M2_HOME/bin:$PATH

  6. Pour que les modifications soient prises en compte dans votre terminal courant, vous devez sourcer le fichier .bash_profile :

    source .bash_profile

Note

Si vous utilisez zsh plutôt que bash, vous devez modifier le fichier .zprofile et non pas le fichier .bash_profile.

Pour Linux

Le plus simple est d’utiliser votre gestionnaire de packages pour installer maven. Pour une distribution basée sur Debian, vous pouvez taper la commande :

sudo apt install maven

Tester l’installation

Ouvrez une fenêtre de terminal, et déplacez-vous dans le répertoire de votre projet Maven grâce à la commande cd (c’est-à-dire le répertoire qui contient le fichier pom.xml). Une fois positionné dans le répertoire du projet, tapez :

mvn clean package

Astuce

Si vous obtenez une erreur car Maven n’arrive pas à trouver de JDK sur votre disque, vous devez également réaliser les étapes suivantes :

  1. ajoutez la variable d’environnement JAVA_HOME indiquant le répertoire d’installation du JDK. Pour Windows, votre JDK devrait être installé dans le répertoire C:\Program Files\Java (par exemple, C:\Program Files\Java\jdk-11). Pour Linux, vous devriez trouver les différentes versions de Java installées dans le répertoire /usr/lib/jvm/.

  2. fermez et ouvrez à nouveau votre fenêtre de terminal,

  3. déplacez-vous dans le répertoire de votre projet grâce à la commande cd,

  4. relancez la commande Maven.

Notion de cycle de vie¶

Maven découpe la construction du projet en phases. L’ensemble de ces phases décrit le cycle de vie d’un projet. Les phases les plus importantes sont :

  1. validate : valide si le projet est correct

  2. compile : compile les sources du projet

  3. test : exécute les tests unitaires

  4. package : crée un package pour le projet compilé (par exemple un fichier JAR pour un programme Java et un fichier WAR pour une application Web Java)

Note

Pour plus d’info sur le cycle de vie d’un projet Maven, consultez la page :

https://maven.apache.org/guides/introduction/introduction-to-the-lifecycle.html

En ligne de commandes, on invoque Maven depuis le répertoire contenant le fichier pom.xml en précisant la phase que l’on souhaite atteindre :

mvn package

Dans l’exemple ci-dessus, la phase package indique que l’on souhaite construire le livrable (un fichier JAR ou WAR par exemple) dans le répertoires target. La phase package implique de réaliser toutes les phases antérieures du cycle de vie. Par exemple, les phases compile et test qui, respectivement, compile le code source et exécute les tests automatisés seront réalisées avant. Donc Maven garantit que la construction du package se fera à partir des dernières sources compilées et que les tests unitaires automatisés auront été exécutés avec succès.

Il est possible de préciser plusieurs phases à Maven. Par exemple :

mvn clean package

Dans ce cas, Maven va commencer par supprimer tous les fichiers de travail (le contenu du répertoire target) et reconstruire le projet à partir de zéro.

Gestion des plugins dans Maven¶

Maven fonctionne en exécutant une succession de plugins. Les plugins sont téléchargés à la demande et mis en cache sur la machine de développement. Par défaut, chaque phase du cycle de vie est associée à l’exécution de un ou plusieurs plugins. Avec Maven, il est possible d’exécuter un plugin directement à partir de la ligne de commandes. Il est également possible de déclarer explicitement un plugin dans le fichier pom.xml pour spécifier des paramètres de configuration et/ou exécuter ce plugin à une phase précise du cycle de vie.

Note

Par défaut, certains plugins sont déjà configurés pour s’exécuter à certaines phases du cycle de vie. Par exemple, le maven-compiler-plugin est appelé par défaut lors de la phase compile.

Un plugin est un artefact Maven. Donc, il est identifié par un ID de groupe, un ID d’artefact et une version. En ligne de commandes, on sépare l’ID de groupe, l’ID d’artefact et la version avec :. On précise enfin l’objectif (goal) que l’on invoque sur le plugin. Chaque plugin dispose de ses propres objectifs qui sont les actions à réaliser. Il faut donc consulter la documentation de chaque plugin.

Par exemple, pour appeler directement l’objectif (goal) compile du plugin de compilation en version 3.1, on peut écrire :

mvn org.apache.maven.plugins:maven-compiler-plugin:3.1:compile

Astuce

Pour trouver le documentation d’un plugin, il suffit la plupart du temps d’utiliser un moteur de recherche avec les mots-clés : maven plugin suivi du nom du plugin.

Exercice¶

Exécuter un plugin en ligne de commandes

Utiliser le plugin maven-dependency-plugin (groupe ID org.apache.maven.plugins et version 2.8) pour afficher la liste des dépendances du projet.

Astuce

Pour connaître les objectifs (goals) disponible sur un plugin, vous pouvez utiliser l’objectif help. Par exemple, pour obtenir l’aide du maven-compiler-plugin :

mvn org.apache.maven.plugins:maven-compiler-plugin:3.1:help

Astuce

Si le nom du plugin est de la forme maven-quelquechose-plugin et que le groupe ID est org.apache.maven.plugins alors on peut directement donner le nom court du plugin :

mvn quelquechose:help

Par exemple :

mvn dependency:help

Pour déclarer un plugin pour un projet particulier, on peut ajouter dans le fichier pom.xml, une entrée pour ce plugin.

Déclaration d’un plugin dans le fichier pom.xml¶
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
                             http://maven.apache.org/xsd/maven-4.0.0.xsd">
  <modelVersion>4.0.0</modelVersion>
  <groupId>io.github.spoonless</groupId>
  <artifactId>demoapp</artifactId>
  <version>1.0-SNAPSHOT</version>

  <build>
    <plugins>
      <plugin>
        <groupId>org.apache.maven.plugins</groupId>
        <artifactId>maven-compiler-plugin</artifactId>
        <version>3.8.1</version>
        <configuration>
            <source>11</source>
            <target>11</target>
            <encoding>utf-8</encoding>
            <verbose>true</verbose>
        </configuration>
      </plugin>
    </plugins>
  </build>
</project>

Dans l’exemple ci-dessus, on déclare explicitement le maven-compiler-plugin afin de passer des paramètres de configuration pour indiquer la version du langage utilisé pour les sources et pour les fichiers de classes, l’encodage des fichiers et le niveau de détail à afficher.

Association d’un plugin à une phase d’exécution¶

Pour déclencher un plugin lors d’une phase précise, il suffit d’utiliser l’élément <execution/> pour préciser la phase concernée, l’objectif à exécuter et la configuration associée :

Déclaration d’un plugin en associant un objectif à une phase¶
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
                             http://maven.apache.org/xsd/maven-4.0.0.xsd">
  <modelVersion>4.0.0</modelVersion>
  <groupId>io.github.spoonless</groupId>
  <artifactId>demoapp</artifactId>
  <version>1.0-SNAPSHOT</version>

  <build>
    <plugins>
      <plugin>
        <groupId>org.apache.maven.plugins</groupId>
        <artifactId>maven-checkstyle-plugin</artifactId>
        <version>3.0.0</version>
        <executions>
          <execution>
            <id>validate</id>
            <phase>package</phase>
            <goals>
              <goal>check</goal>
            </goals>
            <configuration>
              <encoding>UTF-8</encoding>
              <consoleOutput>true</consoleOutput>
              <failsOnError>true</failsOnError>
            </configuration>
          </execution>
        </executions>
      </plugin>
    </plugins>
  </build>
</project>

Dans l’exemple ci-dessous, l’objectif check du maven-checkstyle-plugin est lancé automatiquement lors de la phase package. Dans l’élément <execution/> on définit la phase concernée, l’objectif (goal) à lancer pour ce plugin et éventuellement une configuration. Notez que l’élément <id/> est obligatoire mais ne sert qu’à nommer de manière unique chaque exécution déclarée dans le fichier pom.xml.

Exercice à rendre¶

Utiliser le plugin JaCoCo (exercice à rendre)

JaCoCo est un projet pour la production de rapport de couverture de code par les tests. Configurez votre projet Maven pour utiliser le plugin Maven JaCoCo. Les informations pour le plugin :

  • group ID : org.jacoco

  • artifact ID : jacoco-maven-plugin

  • version : 0.8.5

Pour produire un rapport, il est nécessaire de lancer l’objectif jacoco:prepare-agent, puis la phase test et enfin l’objectif jacoco:report. Vous pouvez associer l’objectif jacoco:prepare-agent à la phase process-test-classes et l’objectif jacoco:report à la phase test.

Astuce

Le rapport JaCoCo est disponible dans le répertoire target/site/jacoco/.

Rendu du projet

Pour partager votre projet, créez un dépôt Git sur https://github.com ou sur https://gitlab.com, partagez les sources du projet pizza-spring avec vos modifications grâce à ce dépôt distant. Transmettez l’adresse de votre dépôt Git par mail à : david.gayerie@reseau-cd.net

Pensez à créer à un dépôt public afin qu’il soit accessible facilement.

Déploiement automatisé dans Tomcat¶

Le serveur Tomcat intègre un gestionnaire d’application appelé le manager.

Astuce

Le manager est simplement une application Web intégrée par défaut dans le serveur.

On peut accéder à ce gestionnaire à travers une application HTML pour les administrateurs ou à travers un échange de commandes pour un processus automatisé. Lorsque vous démarrez Tomcat sur votre machine, l’interface HTML du gestionnaire est accessible à l’adresse :

http://localhost:8080/manager

L’application pour un processus automatisé est accessible à l’adresse :

http://localhost:8080/manager/text

Pour des raisons de sécurité, l’accès au gestionnaire est contrôlé par une authentification login/mot de passe. Par défaut, aucun compte n’a le droit de se connecter à cette application.

Avertissement

Vous ne pouvez pas utiliser le serveur Tomcat installé dans votre environnement Eclipse. En effet, il s’agit d’une version allégée qui n’inclut pas le gestionnaire d’application. Pour la suite du chapitre, nous allons configurer et lancer directement le serveur Tomcat.

Pour démarrer Tomcat, vous devez aller dans le répertoire d’installation de Tomcat. Dans le répertoire bin, vous trouverez le script startup.bat (ou startup.sh selon votre système d’exploitation). Double-cliquez sur ce fichier pour démarrer le serveur.

Si vous voulez en savoir plus sur le comportement du serveur au démarrage (notamment en cas de problèmes), vous pouvez consulter les logs du serveur dans le fichier logs/catalina.out.

Un problème habituel sur les machines de développement et le fait que Tomcat ne peut pas écouter sur les ports nécessaires car ils sont déjà utilisés par d’autres services. Dans ce cas vous aurez une erreur du type Error address already in use et le serveur s’arrêtera. Par défaut, Tomcat écoute sur 3 ports au lancement :

  • le port 8005 appelé le port de shutdown (utilisé pour envoyer la commande d’arrêt au serveur),

  • le port 8009 appelé le port AJP (utilisé pour faire communiquer Tomcat avec un serveur HTTP en front-end)

  • le port 8080 appelé le port HTTP (utilisé pour répondre aux requêtes utilisateur)

Si un de ces trois ports est déjà utilisé par un service sur votre machine, Tomcat ne démarrera pas correctement et ne sera donc pas accessible. Pour modifier ces ports, vous devez modifier la configuration du serveur située dans le fichier conf/server.xml. Recherchez dans ce fichier les occurrences du port que vous voulez modifier et mettez une autre valeur.

Exercice¶

Créer un compte pour accéder au gestionnaire d’application de Tomcat

Pour ajouter un utilisateur dans Tomcat, il faut éditer le fichier conf/tomcat-users.xml depuis le répertoire d’installation de Tomcat. Assurez-vous avant que le serveur est bien arrêté.

Dans ce fichier, on peut ajouter des éléments <user /> pour déclarer un compte avec son mot de passe et les rôles qu’il possède dans le serveur. Pour être autorisé à accéder à l’interface HTML du gestionnaire, un utilisateur doit avoir le rôle manager-gui. Pour être autorisé à accéder à l’interface de déploiement automatisé du gestionnaire, un utilisateur doit avoir le rôle manager-script. Par exemple :

<role rolename="manager-script"/>
<role rolename="manager-gui"/>
<user username="deployer" password="azerty" roles="manager-script,manager-gui"/>

Ajoutez un compte utilisateur ayant les rôles nécessaires, démarrez votre serveur Tomcat et accédez à l’interface HTML du gestionnaire :

http://localhost:8080/manager

Exercice à rendre¶

Déploiement d’une application Web Java avec Maven (exercice à rendre)

Pour cet exercice, nous allons utiliser le plugin org.apache.tomcat.maven:tomcat7-maven-plugin.

Configurez le plugin Maven dans le fichier pom.xml de votre projet pour réaliser le déploiement d’une application Web dans Tomcat après la création du package.

Important

Prenez le temps de consulter la documentation du goal deploy, notamment pour connaître les paramètres que vous pouvez utiliser dans la balise <configuration>. La documentation est consultable à cette adresse :

https://tomcat.apache.org/maven-plugin-2.0/tomcat7-maven-plugin/deploy-mojo.html

Pour réaliser cet exercice, vous avez impérativement besoin d’un couple login/mon de passe pour que le plugin puisse se connecter au gestionnaire de déploiement (Cf. exercice précédent).

Rendu du projet

Faites évoluer votre projet et utilisez la commande push dans Git pour partager vos nouvelles modifications dans votre dépôt distant.

Export des identifiants du serveur¶

Il est problématique d’écrire explicitement le login et le mot de passe du compte utilisateur pour réaliser le déploiement automatisé dans Tomcat. En effet, le fichier pom.xml est souvent partagé au sein de l’équipe, voire au-delà. Avec Maven, il est possible de stocker certaines informations dans le fichier settings.xml. Ce fichier doit être stocké de le répertoire .m2 dans le répertoire de votre compte utilisateur. Ainsi, il est possible de conserver des informations propres à l’utilisateur et non pas au projet.

Le contenu minimal du fichier settings.xml est :

<settings xmlns="http://maven.apache.org/SETTINGS/1.0.0"
      xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
      xsi:schemaLocation="http://maven.apache.org/SETTINGS/1.0.0
                          https://maven.apache.org/xsd/settings-1.0.0.xsd">
</settings>

Ce fichier peut contenir un élément <servers/> qui peut lui-même contenir un ou des éléments <server/>. Le lien entre le serveur déclaré dans le fichier pom.xml et la configuration déclarée dans le fichier settings.xml se fait grâce à un identifiant unique du serveur (décidé par les développeurs).

Pour plus d’information, vous pouvez consulter la documentation du fichier settings.xml :

http://maven.apache.org/settings.html

Exercice bonus¶

Externalisation de la configuration du serveur

Reprenez votre projet Web et externalisez le username et le password dans le fichier settings.xml pour que ces informations n’apparaissent plus dans le fichier pom.xml du projet.
